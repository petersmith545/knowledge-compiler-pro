# 知识编译规则

当用户说「编译 <文件路径>」时，执行以下步骤：

## 第一步：读取原文

读取 `raw/` 下的指定文件，理解内容。

## 第二步：三步编译法

### 步骤1：浓缩
- 用剃刀法则：删掉这条信息会影响理解吗？不会就删
- 输出：核心结论（不超过3条）+ 支撑每条结论的关键证据
- 格式：每条结论一行，证据缩进在下方

### 步骤2：质疑
针对每条核心结论，回答：
1. 这个结论依赖哪些前提假设？
2. 如果这些前提不成立（换行业/换市场/换规模），结论还成立吗？
3. 作者的数据来源可靠吗？样本量、时间范围、地域限制？
4. 有没有作者没提到的反例或边界条件？

### 步骤3：对标
1. 其他领域有没有类似现象？（技术↔商业↔管理↔科学）
2. 这个知识可以迁移到哪些场景？
3. 如果有跨域关联，创建或更新对应的概念条目

## 第三步：产出文件

1. 在 `wiki/summaries/` 写编译摘要（文件名 = 原始文件名 + 编译日期）
2. 在 `wiki/concepts/` 创建或更新相关概念条目
   - 如果概念已存在，合并新信息，标注来源差异
   - 如果两篇文章观点矛盾，在条目中显式标记冲突
3. 更新 `wiki/indexes/index.md`（追加来源 + 关联概念）
4. 更新 `wiki/indexes/log.md`（追加操作记录）

## 概念条目模板


concept: {概念名}sources: [{来源文章1}, {来源文章2}]last_updated: {日期}
{概念名}

定义
关键数据点（附来源）

前提与局限性
冲突标记（如有）

关联概念
## 禁止事项
- 不要修改 `raw/` 下的任何文件
- 不要在编译过程中跳过质疑步骤
- 不要忽略观点矛盾——必须显式标记
